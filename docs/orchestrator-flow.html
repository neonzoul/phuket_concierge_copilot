Part 1 · For Engineers

What this is

Every guest message goes through one orchestrator function, processMessage(), which runs a fixed pipeline of small deterministic agents in order, enforces a timeout/retry/fallback around each one, and logs every attempt. The pipeline decides which of four states the guest sees — ANSWER, CONFIRM, HUMAN, or UNKNOWN — and nothing in this pipeline is allowed to fabricate a fact, confirm a booking, or give medical/legal advice. Today every agent is a hand-written deterministic function (keyword matching, rule lookups) — none of them call an LLM, even though the plumbing for that exists.

The full flow

    G -- "pass" --> R_ANSWER(["ANSWER\nresponse_text + cited_knowledge_ids"])
    G -- "fail" --> R_UNKNOWN

Why safety-guard runs before retrieval

The safety-guard agent is checked first, against the whole raw message, before retrieval or classification even run. This means a high-risk message (e.g. "I have high blood pressure, is detox safe?") never gets a chance to accidentally match an unrelated FAQ — safety short-circuits the entire rest of the pipeline immediately.

Why the classifier checks services before FAQs

Gotcha: the classifier (packages/agents/classifier/src/index.ts) checks for a service-menu keyword match before checking the retrieval agent's FAQ match. This is deliberate — a message like "can I book the morning yoga class?" would also keyword-overlap a wellness FAQ, but a booking-shaped request must become CONFIRM, never a free ANSWER, because nothing in this system is allowed to autonomously confirm a booking. The tradeoff: a genuine FAQ that happens to share a generic word with a service name (e.g. "How long from the airport?" sharing "airport" with the "Airport Transfer" service) can get misrouted to CONFIRM instead of ANSWER. This class of bug is why SERVICE_HAYSTACK_STOPWORDS exists — every time a false collision like this one is found, the shared generic word gets added there rather than reworking the whole matching strategy (see Part 2's Agent reference below for the current list).

What each state actually means to the guest

StateWhat happensCan it ever say "confirmed" / "booked"?
ANSWERReturns a knowledge item's verified answer verbatim, after a citation check passes.No — it can only restate a pre-verified fact.
CONFIRMCreates a pending_confirmation request and tells the guest what's still missing.No — explicitly says staff will confirm.
HUMANCreates an open handoff with prepared context; guest gets a fixed apology template.No — never gives advice, always defers to staff.
UNKNOWNFixed "I don't have confirmed information" template. No knowledge item, no service match, or verification failed.No — this is the deliberate no-guess fallback.

Two ways to end up in UNKNOWN

UNKNOWN isn't just "the classifier gave up." It's also the safety net after a verification failure: if guest-response generates an answer citing a knowledge item that wasn't actually in the retrieved set, or the answer text contains a forbidden phrase (e.g. "guaranteed", "100% safe"), the verifier fails it and the orchestrator downgrades the whole response to UNKNOWN — the guest never sees the unverified answer, even for a split second.

Timeouts, retries, fallbacks

Every agent call goes through runAgent() (packages/agent-runtime/src/runAgent.ts), which retries up to options.retries times on error, and on final failure calls options.fallback() instead of throwing — so a broken agent degrades to a safe default rather than crashing the request. Every attempt (success, error, or fallback) is written to the event log. These per-agent timeout/retry/fallback values live in apps/api/src/server.ts today (passed into createOrchestrator()), not hardcoded inside any agent file.

Entry point

Body: { message: string; guestId?: string; guestName?: string; conversationId?: string; providedFields?: Record<string, unknown> }

Handled in apps/api/src/routes/index.ts:20-30. The handler's only job is calling orchestrator.processMessage({...req.body, propertyContext}) and, based on the result state, writing to requestsRepo (CONFIRM) or handoffsRepo (HUMAN). No orchestration or business logic lives in the route handler itself.

createOrchestrator(timeouts) — packages/orchestration/src/orchestrator.ts

Returns { processMessage(input): Promise<ProcessMessageResult> }. timeouts is an AgentTimeouts object with one RunAgentOptions per agent ({ timeoutMs, retries, fallback }), supplied by the caller (apps/api/src/server.ts) — never hardcoded inside the orchestrator or any agent.

type ProcessMessageResult =
  | { state: "ANSWER"; responseText: string; citedKnowledgeIds: string[] }
  | { state: "UNKNOWN"; responseText: string }
  | { state: "CONFIRM"; responseText: string; request: RequestObject }
  | { state: "HUMAN"; responseText: string; handoff: HandoffObject };

Pipeline order (exact)

  1. safetyGuardAgent.execute({ message, propertyContext }) → SafetyGuardOutput. If matched && behavior_state === "HUMAN", call humanHandoffAgent and return { state: "HUMAN" } immediately — retrieval/classifier never run.
  2. retrievalAgent.execute({ message, propertyContext }) → RetrievalOutput (always runs if safety-guard didn't short-circuit).
  3. classifierAgent.execute({ message, retrieval, propertyContext }) → ClassifierOutput. Branches on classification.behavior_state:
    • UNKNOWN → return fixed template, no further agent calls.
    • CONFIRM → requestCaptureAgent.execute({ serviceId: classification.intent, guestId, guestName, providedFields, sourceMessageId: traceId, propertyContext }) → RequestObject.
    • HUMAN → humanHandoffAgent.execute(...), with a synthetic safetyResult built from the classifier's intent/reason (hardcoded risk_level: "MEDIUM", handoff_team: "guest_relations" — this is the fallback branch for HUMAN states the safety-guard's rule table didn't catch).
    • ANSWER (default) → guestResponseAgent.execute({ knowledgeItems: retrieval.matched_items }) → verifierAgent.execute({ response, citedItems: retrieval.matched_items }). If !verification.pass, downgrade to UNKNOWN.

runAgent() contract — packages/agent-runtime/src/runAgent.ts

runAgent<TInput, TOutput>(agent: Agent<TInput,TOutput>, input: TInput, context: AgentExecutionContext, options: RunAgentOptions): Promise<TOutput>

interface RunAgentOptions { timeoutMs: number; retries: number; fallback: () => unknown }
  • Races agent.execute() against a setTimeout rejection at timeoutMs.
  • Retries up to options.retries additional times on any thrown error (including timeout).
  • On exhausted retries, calls options.fallback() and returns its result — never throws out of runAgent itself.
  • Writes one EventLogEntry per attempt via appendEvent(), event_type one of agent.success / agent.error / agent.fallback, keyed by context.traceId.

Invariants (do not violate when editing this area)

  • No agent calls another agent directly — only the orchestrator calls agents, and only through runAgent.
  • No agent file hardcodes a model name, prompt path, timeout, or retry count — those come from config/agents.registry.ts (registry) and the timeouts object threaded into createOrchestrator (runtime behavior); today's deterministic agents don't read the registry at all since they don't call an LLM, but the registry is still the intended single source of truth once any agent flips to provider: "anthropic".
  • No property-specific fact appears in agent source — everything comes from propertyContext (loaded from contexts/{slug}/*.json).
  • HUMAN is not equivalent to URGENT — priority is a separate axis set by the matched rule/team, never implied by the state alone.
  • guest-response only ever echoes a knowledge item's existing verified_answer — it does not paraphrase or generate new text today.

Known gap (see spec/implementation-plan.md)

39/40 acceptance-suite cases pass as of this doc's commit — see "Acceptance suite fix log" in spec/implementation-plan.md for the full diff-by-diff history of what was fixed and why (a mix of classifier stopword/threshold tuning and content edits to the approved demo data pack, kept in sync with Resources/build-pack/phuket_concierge_copilot_demo_data_pack_v1_0/). The one remaining failure, TRAP-007 ("Can I check in at 8 AM guaranteed?", expects CONFIRM), is left as a deliberate, documented limitation: check is stripped from every service haystack repo-wide (that's what fixes the "checkout" collision above), and no other word in that message is a safe, generalizable match signal without bigram-aware matching — a real scope increase, not a one-line fix. Don't add a single-purpose rule keyed to that exact phrasing; that would overfit the test rather than fix the product.
